## 架构变更：规则引擎拆分

**原架构：**
```
Next.js → Python (全部评估逻辑) → DeepSeek
```

**新架构：**
```
Next.js API Routes (Vercel)
├── 规则引擎（本地）
│   ├── 爽点密度（关键词匹配）
│   ├── 节奏分析（段落长度/对话密度）
│   └── 注水检测（字符串相似度）
└── /api/evaluate
    ├── 输入校验 (null/empty/length/language)
    ├── 并行执行 (Promise.all)
    │   ├── 规则引擎（本地）
    │   └── DeepSeek API 直接调用（openai npm 包）
    │       ├── Hook 强度（LLM）
    │       ├── 章末悬念（LLM）
    │       └── 一致性检查（LLM）
    ├── 结果合并 + 部分评估处理
    └── 数据库存储 (Prisma → PostgreSQL/Supabase)
```

Phase 0 不需要 Python 服务。DeepSeek API 通过 `openai` npm 包直接调用。
Phase 2+ 如果需要重 NLP（spaCy、sentence-transformers），可引入 Python 微服务。

**好处：**
- Phase 0 只需 Vercel 一个部署目标，运维极简
- 减少服务间调用（规则引擎在本地）
- 降低延迟（规则引擎不走网络）
- DeepSeek API 不可用时，规则引擎部分仍可用（返回部分评估）

## 关键变更：LLM 评估一致性

**问题：** 同一章节评估两次可能得到不同分数，摧毁用户信任。

**解决方案（三重保障）：**

1. **temperature=0：** DeepSeek API 调用时设置 temperature=0，最大化确定性
2. **分数钳制：** 所有子分数强制钳制在 0-10 范围内，综合分数四舍五入到小数点后 1 位
3. **方差预算：** 同一章节 + 同一 prompt 版本，多次评估的分数波动应 < 0.5 分。如果超过，视为异常并记录日志

**实现位置：** Next.js API Routes 的 `/api/evaluate`，返回结果前做分数后处理。
