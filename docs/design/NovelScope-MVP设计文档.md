## Problem Statement

中文网文作者（3269万人）大量使用AI辅助创作，但现有工具全部聚焦于"内容生成"（红海，20+竞品），无人系统性地做"质量评估"。作者写完一章后，没有工具能告诉他们：这一章的追读力如何？逻辑有没有崩？节奏拖不拖沓？

创始人自身是网文作者，试过所有主流工具（webnovel-writer、星月写作、蛙蛙写作、笔灵AI），全部放弃。放弃原因不是某个功能不行，而是"综合体验差" — 现有工具都在错误的品类里。

## Demand Evidence

**行为信号（强）：**
- 创始人试过所有主流AI写作工具，全部放弃 — 这是"在找不存在的品类"的典型表现
- 社交媒体上有大量作者对现有AI创作工具表达不满
- 创始人能直接联系到潜在用户（龙的天空论坛、起点作者QQ群、知乎网文话题、贴吧写作吧）

**市场信号（强）：**
- 追读率预测AI：中英文搜索、学术数据库、竞品分析全部确认空白
- 质量评估工具：99%的AI写作工具是"生成导向"，评估导向极度稀缺
- AI工具使用率60-75%，但质量评估场景占比<5%
- 政策环境（《AI生成内容标识办法》）利好"辅助评估"定位

**自述信号（中）：**
- 创始人自述痛点为"体验问题"，非财务损失
- 但行为信号强于自述 — 实际行动比自我描述更能反映真实需求

## Target User & Narrowest Wedge

**核心用户：** 日更连载网文作者 — 每天写5000-10000字，已签约或想签约，追读数据直接影响收入。

**最窄楔子：** 粘贴一章文本 → 获得质量报告（追读力评分 + 逻辑一致性检查 + 节奏分析）。每章评估多次（起草→审查→修改→再审查→发布），日均使用2-4次。

**为什么是这个楔子：**
- 直接验证核心假设：用户是否愿意为"评估"付费
- 不需要用户迁移成本（不需要导入历史章节）
- 单次使用价值明确（一份报告 vs "试试看"）

## Constraints

- **单人开发：** 一个人+AI辅助（Cursor/Claude Code），2-3个月出MVP
- **成本敏感：** 首年运营预算12.62万元，API成本需控制
- **技术栈已定：** Next.js + TypeScript + Tailwind CSS + DeepSeek API（通过 openai npm 包）、PostgreSQL + Prisma（数据库）
- **政策合规：** 定位为"辅助评估"而非"自动生成"，遵守《AI生成内容标识办法》
- **市场窗口：** 追读率预测空白，但窗口不会永远开着

## Premises

1. **网文作者需要"质量评估"而非"内容生成"** — ✅ 竞品分析+用户痛点验证。99%工具做生成，评估品类空白。
2. **"追读力"可以被量化** — ✅ 蓝海确认，技术路线可行（规则引擎+大模型评分混合方案）。
3. **用户愿意为"评估"付费** — ⚠️ 间接推断。创始人确认评估是高频行为（每章多次），支持月订阅模式。但尚未直接验证（没有人实际付过钱）。
4. **一个人+AI可以构建此产品** — ⚠️ 需要严格控制范围。MVP只做评估+极简记忆管理，不做AI创作/商业化分析。

## Cross-Model Perspective

**Claude Subagent 独立审阅（冷读）：**

- **Steelman：** NovelScope 不是在做"更好的写作工具"，而是在创造一个新品类 — "编辑之眼"。把主观的创作技艺变成可量化的指标。
- **关键洞察：** 创始人在找一个不存在的品类，所以无法描述缺什么。现有工具都在错误的品类里。这是品类创造的机会。
- **挑战前提 #3：** "评估可能是低频行为（一本书评估一次），而非高频习惯"。如果低频，月订阅走不通。
- **创始人反驳：** "一章会评估数次"（起草→审查→修改→再审查→发布）。对日更作者来说是每天2-4次的高频行为。反驳成立。
- **原型建议：** 48小时内做一个粘贴→报告的单页应用，验证核心假设。

**综合判断：** Subagent 的"品类创造"洞察有价值 — 你的定位不是"更好的X"，而是"第一个Y"。前提 #3 的挑战被创始人的具体使用场景反驳，但直接验证（有人付钱）仍是 MVP 的首要目标。

## Approaches Considered

### Approach A: 最小验证 — "粘贴即报告"
- 单页应用，粘贴章节→输出质量报告
- 不搞用户系统、不搞记忆管理
- Effort: S（1-2周），Risk: Low
- 优：最快验证核心假设。缺：无粘性，用一次就走

### Approach B: 高频核心 — 评估+极简记忆（推荐）
- 质量评估（追读力+一致性+节奏）+ 极简记忆管理（角色档案+设定约束+上下文注入）
- Effort: M（2-3个月），Risk: Medium
- 优：记忆管理让产品变成"每天打开"的工具，评估+记忆组合=写作工作流核心环节。缺：比A慢

### Approach C: 完整平台 — 文档里全部五模块
- 追读力分析、质量审查、记忆管理、商业化、AI创作
- Effort: XL（6-12个月），Risk: High
- 优：最完整愿景。缺：一个人做6-12个月，没有验证就大量投入

## Recommended Approach

**Approach B: 高频核心 — 评估+极简记忆管理**

选择理由：
1. 记忆管理是评估的基础设施 — 没有记忆，长篇一致性检查就是空话
2. 记忆管理是留存的钩子 — 评估是"付费理由"，记忆是"每天打开的理由"
3. 2-3个月可执行，一个人+AI辅助能搞定
4. 比A有粘性，比C（完整平台）可执行

### Architecture

**Phase 0 架构（单服务，简化）：**

```
用户浏览器
    |
    v
Next.js App (Vercel)
├── 前端 UI (React + Tailwind)
├── API Routes
│   ├── /api/evaluate (评估入口)
│   │   ├── 输入校验 (null/empty/length/language)
│   │   ├── 阶段 1: 规则引擎信号提取（本地）
│   │   │   ├── 爽点密度 → keywordCategories, matchedKeywords, dialogueDensity
│   │   │   ├── 节奏分析 → cv, typeRatio, curve
│   │   │   └── 注水检测 → suspiciousPairs, items
│   │   ├── 阶段 2: 信号注入 LLM prompt 构建
│   │   ├── 阶段 3: LLM 评估（DeepSeek API，输出全部 4 维分数）
│   │   ├── 阶段 4: Guard（clamp + 加权综合分）
│   │   ├── 降级: LLM 失败 → 使用规则引擎参考分数
│   │   └── 数据库存储 (EvaluationReport)
│   └── 用户/章节/报告 CRUD
└── Prisma ORM → PostgreSQL (Supabase, pgbouncer)
    |
    v
DeepSeek API (openai npm 包, temperature=0)
```

**Phase 2+ 架构（可选扩展）：**

Phase 2 如果需要重 NLP（spaCy、sentence-transformers），可引入 Python 微服务。Phase 0 不需要。

**通信方式：** Next.js API Routes 是唯一的后端。规则引擎先执行提取结构化信号，信号注入 LLM prompt 后由 LLM 进行最终评分。DeepSeek API 通过 `openai` npm 包直接调用（DeepSeek 兼容 OpenAI API）。LLM 失败时降级到规则引擎参考分数。

**设计系统：** 所有 UI 实现必须遵循 `DESIGN.md` 中定义的字体、色彩、间距和美学方向。

**LLM 输出处理：** DeepSeek 返回的 JSON 可能包含 markdown 代码块或额外说明。处理流程：strip markdown fences → JSON.parse (try-catch) → Zod schema 验证 → 重试 1 次（失败则返回部分评估）。

### 追读力评分体系（初版）

| 指标 | 评分范围 | 计算方式 | 权重 |
|------|----------|----------|------|
| Hook强度 | 0-10 | LLM 评估：首段是否抓人、是否有悬念、是否引发好奇 | 30% |
| 爽点密度 | 0-10 | LLM 基于规则引擎信号评分：关键词命中分类、对话密度、冲突密度 | 30% |
| 章末悬念 | 0-10 | LLM 评估：章末是否有未解决的冲突、是否有翻页冲动 | 25% |
| 节奏得分 | 0-10 | LLM 基于规则引擎信号评分：CV、动作/对话/描写比例、张力曲线 | 15% |

**综合追读力 = Hook×0.3 + 爽点×0.3 + 悬念×0.25 + 节奏×0.15**

权重为初版，基于"黄金三章"网文写作经验设定。上线后根据用户反馈和实际追读数据校准。

**LLM 评估一致性保障（关键）：**

同一章节评估两次必须得到相近分数，否则用户信任崩塌。三重保障：

1. **temperature=0：** DeepSeek API 调用时设置 temperature=0，最大化确定性输出
2. **分数钳制：** 所有子分数强制钳制在 0-10 范围内，综合分数四舍五入到小数点后 1 位
3. **方差预算：** 同一章节 + 同一 prompt 版本，多次评估的分数波动应 < 0.5 分。超过则视为异常并记录日志

实现位置：Next.js API Routes 的 `/api/evaluate`，返回结果前做分数后处理。

**部分评估处理：**

当 LLM 调用失败时，标记 `isPartial: true`，使用规则引擎的参考分数作为 fallback（climaxScore 和 pacingScore 来自规则引擎，hookScore 和 cliffhangerScore 为 0）。用户看到"部分评估"提示，知道这是降级评估。

### 评估管线（信号注入架构）

架构采用"规则引擎作为信号提取器，LLM 做最终评分"的模式：

```
规则引擎 → 结构化信号 ──→ 注入LLM prompt → LLM出全部4个分 → guard → overallScore
```

| 子评估 | 信号来源 | LLM 角色 | 理由 |
|--------|----------|----------|------|
| Hook强度 | 无规则信号 | LLM 独立评分 | 需要理解语义和叙事技巧，规则引擎无法判断 |
| 爽点密度 | ClimaxAnalyzer → keywordCategories, dialogueDensity, conflictDensity | LLM 基于信号评分 | 规则引擎提供关键词命中等信号，LLM 理解语义上下文后给出更准确的分数 |
| 章末悬念 | 无规则信号 | LLM 独立评分 | 需要判断叙事张力和未解决冲突，规则引擎无法判断 |
| 节奏分析 | PacingAnalyzer → cv, typeRatio, curve | LLM 基于信号评分 | 规则引擎提供段落分类和变异系数，LLM 综合判断节奏质量 |
| 一致性检查 | 无规则信号 | LLM + 角色档案上下文 | 将角色档案和设定约束注入system prompt，让LLM对照检查 |
| 注水检测 | FillerDetector → suspiciousPairs, items | LLM 参考信号判断 | 规则引擎提供相似度数据，LLM 判断是否为有意重复 |

**信号格式化：** 规则引擎输出的结构化数据通过 Prompt 构建模块（`services/prompt/`）格式化为中文提示，注入 LLM 的 system prompt。调整评估逻辑只需改 prompt，不改规则引擎代码。

### 记忆管理（极简版）

**自动提取流程：**
1. 用户粘贴章节文本
2. LLM提取 `{ characters: [{ name, aliases, traits, relationships }] }`
3. 展示给用户，用户可编辑修正
4. 存入 CharacterProfile 表

**上下文注入：**
- 评估时，从当前小说的所有 CharacterProfile 和 SettingConstraint 中加载相关条目
- 注入 LLM 的 system prompt："以下是本小说已建立的角色和设定，请在评估时对照检查..."
- 限制上下文长度：最多加载最近20个角色 + 所有core设定约束

**设定约束：**
- 用户可手动添加核心设定（如"主角是火属性修炼者"、"世界观是修仙体系"）
- 标记为 `isCore: true` 的约束在一致性检查中不可违背
- LLM评估时会在报告中标注"设定违背"问题

### Edge Cases

| 场景 | 处理方式 |
|------|----------|
| chapterText 为 null/undefined | 400 错误，提示"请输入章节文本" |
| chapterText 为空字符串 "" | 400 错误，提示"请输入章节文本" |
| 章节文本 < 100字 | 400 错误，提示"文本不足100字，无法评估。建议至少500字以获得有意义的报告。" |
| 章节文本 > 50000字 | 400 错误，提示"文本过长，请分段提交" |
| 非中文文本 | 400 错误，检测语言，非中文提示"目前仅支持中文网文" |
| LLM API 超时 | 指数退避重试（1s → 2s），最多 2 次。仍失败返回部分评估结果（规则引擎部分） |
| LLM API 返回 429（限流） | 退避重试（2s → 4s），最多 2 次。仍失败返回部分评估结果 |
| LLM 返回空响应/拒绝回答 | 不重试，直接返回部分评估结果 + 提示"LLM 服务暂时不可用" |
| LLM 返回格式异常（非法 JSON） | 重试 1 次，仍失败返回部分评估结果 |
| LLM 返回格式不符（缺少字段） | 重试 1 次，仍失败返回部分评估结果 |
| DeepSeek API 不可用 | 重试 2 次（指数退避），仍失败返回部分评估结果（仅规则引擎） |
| 无记忆上下文的首次评估 | 跳过一致性检查（无角色档案可对照），仅输出追读力+节奏分析 |
| 同一章节多次评估 | 创建新 EvaluationReport 记录，保留历史，用户可对比 |
| LLM 评分方差 > 0.5 | 记录异常日志，但仍然返回结果（不阻断用户） |

### Auth

- **MVP阶段：** 邮箱 + 验证码登录（magic link），无需密码
- **用户数据：** 仅存储邮箱、昵称、创建时间
- **理由：** 最低摩擦的登录方式，不要求用户设置密码，适合快速验证阶段

### 情绪设计（报告语气）

**核心原则：优势先行 + 建设性语气**

报告的语气是"编辑给作者的反馈"，不是"考试成绩单"。

| 场景 | 错误示范 | 正确示范 |
|------|----------|----------|
| 综合分数 4.2 | "综合追读力：4.2/10" | "你的开头Hook很强（8.1分），但爽点密度有提升空间（4.2分）。以下是具体建议：" |
| 一致性问题 | "角色设定冲突" | "第3段的角色表现与前文设定有差异，建议调整" |
| 注水检测 | "第5段注水" | "第5段与第3段内容相似，精简后节奏更紧凑" |

**规则：**
- 永远不裸露低于 5 分的数字，必须附带上下文和建议
- 先说亮点，再说改进点
- 用"建议"替代"问题"，用"可以提升"替代"不好"
