# Issue #16: 报告信息架构重排

## What to build

当前 ReportCard 的信息层级：综合分（主角）→ 四个子分数 → 亮点 → 建议 → 注水 → 节奏曲线。这与"编辑的反馈"定位矛盾——优点和改进建议应该优先于分数可视化。

重新排列报告的信息架构：**亮点（优点优先）→ 改进建议（按严重度分级）→ 四维雷达图 → 节奏曲线 → 注水检测 → 一致性检查**。

同时优化 LLM prompt，让模型产出更高质量、更具体的改进建议（按严重度分级、附带可操作的具体方向）。

## Acceptance criteria

- [x] ReportCard 信息层级重排：
  - 区域 1：亮点分析（优点优先，正面反馈）
  - 区域 2：改进建议（按严重度分级：🔴 关键 / 🟡 建议 / 🔵 观察，每条建议附带具体方向）
  - 区域 3：四维雷达图（可视化，降级为辅助信息）
  - 区域 4：节奏曲线 SVG
  - 区域 5：注水检测列表
  - 区域 6：一致性检查（如有问题）
  - 区域 7：Token 用量 + 成本（小字，不干扰主视觉）
- [x] LLM prompt 更新：改进建议要求按严重度分级，每条包含"问题定位 + 严重度 + 具体改进方向"
- [x] LLM Zod schema 扩展：suggestions 从 `string[]` 升级为 `{ severity: "critical"|"warning"|"info", location: string, issue: string, direction: string }[]`
- [x] ReportCard 各区域有清晰的视觉分区（section 标题 + 分隔），但不制造"卡片套卡片"的冗余嵌套
- [x] 空状态处理：无亮点时显示"规则引擎未检测到显著亮点"，无建议时显示"未发现明显问题"
- [x] 后端测试：Zod schema 向后兼容旧格式 string[]
- [x] 前端测试：各区域渲染覆盖 + 空状态覆盖
- [x] 前端测试：建议按严重度排序（critical > warning > info）
